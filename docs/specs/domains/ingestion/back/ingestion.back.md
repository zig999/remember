## 2. Data Model

> Exact database types as defined in `migrations/0001_init.sql` plus the v1.5.0 additive migration `migrations/0002_original_input.sql`. This revision **introduces no new table and no new index** ŌĆö but it adds ONE new nullable column on a table owned by this domain: `raw_information.original_input text NULL` (verbatim chat-turn user input on the directed-ingestion path; see BR-34 step 2bis). No other column / table / index changes. The ingestion domain still owns six tables: `raw_information`, `raw_chunk`, `llm_run`, `tool_call`, `information_fragment`, `fragment_source`. The new behavior introduced by previous revisions (entity-resolution at write time, graph consolidation at write time) reads and writes the existing tables of the graph layer (`knowledge_node`, `node_alias`, `node_attribute`, `knowledge_link`, `provenance`, `entity_match_review`) ŌĆö those rows are written from inside the ingestion transaction; the tables themselves are owned by the future `graph-consolidation` and `entity-resolution` domains (┬¦7 of `ingestion.spec.md`).

### Indexes (unchanged in this revision)

All indexes consumed by the new resolver/consolidator already exist in `migrations/0001_init.sql`. **`raw_information.original_input` is intentionally NOT indexed in v1** ŌĆö there is no full-text (`tsvector`), no trigram, and no btree on the new column. Rationale: the column is a passive audit / provenance surface (consumed at read time by the query-retrieval provenance walk; `NodeProvenanceChain` displays the verbatim turn when present); the existing search surface against fragments (`information_fragment.text` + full-text indexes already in `0001_init.sql`) covers content-retrieval needs. Adding an index is a future evolution if a "search by operator turn" use case materialises; out of scope for v1.5.0.

| knowledge_link | (source_node_id, link_type_id, target_node_id) WHERE vigent | partial UNIQUE `knowledge_link_current_dup_guard` | **The dup-guard is scoped by `(source, link_type, target)` ŌĆö `valid_from` is NOT part of the key.** Without consolidation, a second identical assertion would hit this constraint. BR-27 turns that case into `consolidated` instead, including the multi-current branch where the second assertion's `valid_from` differs from the vigent row (typically because the new assertion's basis is the `received` fallback of a later document ŌĆö see BR-27 Re-affirmation sub-cases). |
| node_attribute | (node_id, attribute_key_id, value) WHERE vigent | partial UNIQUE `node_attribute_current_dup_guard` | Same as above for attributes. |

### Relationships (unchanged in this revision)

## 3. Business Rules (BR)

> Every BR references at least one UC of `ingestion.spec.md`. This section translates each business rule into the validation layer that enforces it and the error code returned on violation. Rule wording is condensed; canonical wording lives in `ingestion.spec.md` ┬¦4.

### BR-01 -- Content hash is the idempotency anchor
**Related UC:** UC-01
**Where to validate:** service (`ingestion.service.ingestRawInformation`) computes `sha256(content)` after the Zod parse and before the transaction opens. Format constraint (`^[0-9a-f]{64}$`) is enforced by the DB `CHECK` on `raw_information.content_hash`.
**Description:** `content_hash = sha256(content)`, hex, lowercase, 64 chars. The DB `UNIQUE (content_hash)` is what UC-01 alt 4a turns into `outcome = "noop_existing"`. **The new `original_input` column (v1.5.0 / BR-34 step 2bis) is NOT a participant** ŌĆö the hash is computed over `content` only, and `content` continues to be the synthesised fragments-plus-nonce block on the directed-ingestion path. Two directed calls with identical `fragments[]` and `source_label` but different `original_input` therefore still produce different `content_hash` values (because the per-call nonce is part of `content`, not because of `original_input`); two non-directed calls with identical `content` and different `original_input` (impossible by construction ŌĆö `original_input` is only set on the directed path) would collide on `content_hash` exactly as today.
**Error returned:** No direct error ŌĆö collision is a business outcome (`200 noop_existing`), not a 409.

### BR-02 -- Source content is immutable
**Related UC:** UC-01, UC-02
**Where to validate:** repository ŌĆö the ingestion module exposes no `UPDATE raw_information ...` query at all. The only writer that exists in the whole system targeting `raw_information.content` belongs to the future compliance module (`compliance_delete`).
**Description:** After insert, `raw_information.{content, metadata, received_at}` is never modified by any code path of this domain.
**Error returned:** Not applicable ŌĆö enforced by absence of code (no error path needed).

### BR-03 -- Chunking is deterministic and versioned
**Related UC:** UC-01
**Where to validate:** Vitest unit test on `chunkV1` ŌĆö running `chunkV1(content, sourceType)` twice produces strictly equal outputs. `chunking_version` is hardcoded to `'v1'`.
**Description:** Same `(content, chunking_version)` ŌåÆ same chunks.
**Error returned:** Not applicable at request time.

### BR-04 -- Chunk algorithm constants are fixed
**Related UC:** UC-01
**Where to validate:** `modules/ingestion/chunker/config.ts` ŌĆö `CHUNK_TARGET=[1500,2000]`, `CHUNK_HARD_MAX=4000`, `READING_TAIL=200` (A22).
**Description:** The three constants are not configurable per request.
**Error returned:** Not applicable.

### BR-05 -- Chunk offsets are 0-based, semi-open, in Unicode code points
**Related UC:** UC-01, UC-03
**Where to validate:** chunker implementation ŌĆö iterate via `[...str]`. DB `CHECK (offset_end > offset_start)` is the safety net.
**Description:** `offset_start >= 0`, `offset_end > offset_start`, both in code points of the original `content`.
**Error returned:** DB `CHECK` violation ŌåÆ `500 SYSTEM_INTERNAL_ERROR`.

### BR-06 -- Hard boundaries close the current chunk
**Related UC:** UC-01
**Where to validate:** chunker ŌĆö per-`source_type` dispatch table.
**Description:** Hard boundaries are mandatory closures; the chunker never overlaps chunks; `READING_TAIL` is not persisted.
**Error returned:** Not applicable.

### BR-07 -- Oversize blocks fall back to sentence split
**Related UC:** UC-01
**Where to validate:** chunker ŌĆö `Intl.Segmenter('pt', { granularity: 'sentence' })`. Code blocks and tables are exempt.
**Description:** Sentence-level split is a last-resort fallback; structural blocks are preserved intact.
**Error returned:** Not applicable.

### BR-08 -- Run idempotency key composition
**Related UC:** UC-01
**Where to validate:** service ŌĆö `idempotencyKey = sha256(utf8(content_hash) Ōłź utf8(prompt_version) Ōłź utf8(model) Ōłź utf8(chunking_version))`, no separator. Hex lowercase 64 chars.
**Description:** Bumping any of the four operands yields a different key. **`original_input` (v1.5.0) is NOT one of the four operands** ŌĆö it does not flow into the idempotency_key (transitively, through `content_hash`, or otherwise). Two ingestions with the same `content` and different `original_input` therefore have the SAME `idempotency_key` and BR-09 reduces the second to `noop_existing`; the first call's `original_input` is preserved verbatim on the `raw_information` row, the second call's `original_input` is **NOT** persisted (no UPDATE ŌĆö BR-02 immutability is preserved; the directed path's per-call nonce in `content` normally prevents this collision from occurring in practice).
**Error returned:** Not applicable ŌĆö collisions translate into `200 noop_existing`.

### BR-09 -- Re-ingestion is a no-op on the live path
**Related UC:** UC-01 alt 4a
**Where to validate:** service. Catch SQLSTATE `23505` on `raw_information_content_hash_key`, re-read, recompute `idempotency_key`, look up the existing `llm_run`. If the run is found, return `200 noop_existing` regardless of its `status` (failed runs still surface via this 200 path; the caller is expected to invoke `retryLlmRun` next).
**Description:** No new rows are written in the no-op path.
**Error returned:** None ŌĆö 200 success.

### BR-10 -- Retry reopens the same LLMRun row
**Related UC:** UC-06
**Where to validate:** service (`ingestion.service.retryLlmRun`). One transaction: `UPDATE llm_run SET status='running', attempts = attempts+1, finished_at = NULL WHERE id = $1 AND status = 'failed' RETURNING *`. In the same transaction, flip orphan `proposed` fragments to `rejected` (no row in `provenance`).
**Description:** A new `llm_run` row with the same `idempotency_key` is never created.
**Error returned:** `409 BUSINESS_RUN_NOT_RETRYABLE` if not failed.

### BR-11 -- Only failed runs are retryable
**Related UC:** UC-06
**Where to validate:** service ŌĆö conditional `UPDATE ... WHERE status = 'failed'` plus a pre-read to distinguish 404 from 409.
**Description:** Re-running a `running` or `completed` run is rejected.
**Error returned:** HTTP 409 -- error.code: `BUSINESS_RUN_NOT_RETRYABLE`.

### BR-12 -- Run summary is derived, never stored
**Related UC:** UC-04
**Where to validate:** repository ŌĆö aggregate `tool_call.validation_outcome` for the run.
**Description:** No `summary_*` columns on `llm_run`.
**Error returned:** None ŌĆö read-only.

### BR-13 -- Layered validation order is fixed
**Related UC:** UC-08, UC-09, UC-10, UC-11
**Where to validate:** service. Each `propose_*` service function invokes the 5 layers in order: `structural ŌåÆ graph rules ŌåÆ temporal ŌåÆ confidence ŌåÆ anti-hallucination`. The handler catches `ValidationFailure` and returns the matching MCP error envelope; the `tool_call` row is persisted with `validation_outcome = 'rejected'`.
**Description:** Rejection is a business result, not an exception; layers must not short-circuit out of order.
**Error returned:** See BR-14..BR-17.

### BR-14 -- Structural failures map to VALIDATION_* / BUSINESS_UNKNOWN_* / RESOURCE_NOT_FOUND
> P2.1 canonical taxonomy (2026-07-02). The former ┬¦14 short codes (`STRUCTURAL_INVALID` / `UNKNOWN_TYPE` / `NOT_FOUND`) are deprecated on both transports.

**Related UC:** UC-08, UC-09, UC-10, UC-11
**Where to validate:** service / structural layer.
**Description:**
- Missing / typed fields / length or range violation / cross-table compatibility mismatch ŌåÆ `VALIDATION_REQUIRED_FIELD` | `VALIDATION_INVALID_FORMAT` | `VALIDATION_OUT_OF_RANGE` (Zod-discriminated, same discrimination REST already uses).
- Catalog miss on `node_type` ŌåÆ `BUSINESS_UNKNOWN_NODE_TYPE`; on `link_type` ŌåÆ `BUSINESS_UNKNOWN_LINK_TYPE`; on `key` ŌåÆ `BUSINESS_UNKNOWN_ATTRIBUTE_KEY` (three replacement codes, discriminated by which catalog table the missing name belongs to).
- Referenced row not found (`chunk_id` / `fragment_id` / `node_id`) ŌåÆ `RESOURCE_NOT_FOUND`.
- Anti-hallucination layer 5 (BR-18) emits `VALIDATION_INVALID_FORMAT` ŌĆö layer 5 rejections roll up under the `structural` rejected-by-layer bucket alongside layer 1.
**Error returned:** MCP envelope `{ ok: false, error: { code } }` ŌĆö the `code` is one of the P2.1 namespaced codes above, byte-identical to the REST wire code. `tool_call.validation_outcome = 'rejected'`. Parity is a CI gate ŌĆö BR-13..BR-18 parity assertion (companion to compliance BR-14, curation BR-32, knowledge-graph TC-04).

### BR-17 -- Confidence routing
**Related UC:** UC-10, UC-11
**Where to validate:** service / confidence layer (A13). Constants in `modules/ingestion/validation/confidence.ts`: `CONFIDENCE_FLOOR = 0.40`, `CONFIDENCE_UNCERTAIN_UPPER = 0.75`.

### BR-18 -- Every accepted assertion has provenance
**Related UC:** UC-10, UC-11

**Error returned:** MCP envelope `{ ok: false, error.code: "VALIDATION_INVALID_FORMAT" }` ŌĆö P2.1 namespaced; layer-5 rejections roll up into the `structural` rejected-by-layer bucket (BR-14 note).

### BR-19 -- One transaction per tool call
**Related UC:** UC-08, UC-09, UC-10, UC-11

**Description:** A run of N tool calls that fails on call K keeps the K-1 already-accepted units.
**Error returned:** Not applicable.

**Error returned:** Not applicable.

### BR-21 -- `ingest` toolset only operates inside an active LLMRun (transport-agnostic, arg-based run binding)
**Related UC:** UC-08, UC-09, UC-10, UC-11
**Where to validate:** service ŌĆö every `propose_*` service function takes a `runContext = { llmRunId, rawInformationId }` argument and verifies at entry (`assertRunIsRunning`) that `llm_run.status = 'running'` (SQLSTATE-safe read; if not running, throws `ValidationFailure('BUSINESS_RUN_NOT_RUNNING')` when the row exists but is not `running`, or `ValidationFailure('RESOURCE_NOT_FOUND')` when the row does not exist at all ŌĆö P2.1 namespaced; the pre-P2.1 blanket `STRUCTURAL_INVALID` mapping is retired). The run-id binding is per-call ŌĆö there is no ambient session header, no per-session factory, no transport-level toolset gating:
- **MCP transport (stateless single-shape, via `mountMcpEndpoint`):** the four `ingest` tools are **always listed** by the MCP `Server`'s `tools/list` handler, regardless of any per-call state. The MCP-facing Zod schema of each `propose_*` tool extends the business DTO with `llm_run_id: z.string().min(1)` (Option B ŌĆö argument binding). The thin MCP handler (`modules/ingestion/mcp/propose-*.handler.ts`) reads `llm_run_id` from the tool **args**, builds the `runContext`, and calls the service. A call whose `llm_run_id` is missing or malformed returns the MCP 2025-06-18 `content` / `isError: true` shape carrying `{ ok: false, error: { code: "VALIDATION_REQUIRED_FIELD" | "VALIDATION_INVALID_FORMAT", message } }` (P2.1 Zod-discriminated); a call whose `llm_run_id` resolves to no row returns `{ ok: false, error.code: "RESOURCE_NOT_FOUND", ... }`; a call whose ambient run exists but `status != 'running'` returns `{ ok: false, error.code: "BUSINESS_RUN_NOT_RUNNING", ... }` (byte-identical to the REST 409 code). HTTP `200` from the SDK kernel ŌĆö business errors are NOT HTTP errors. The handler is **always reached** ŌĆö there is no pre-handler rejection path. **Local stdio transport (Emenda v7.5):** the same four `propose_*` tools (plus `ingest_document`, BR-30) are ALSO registered on the local stdio transport (`backend/src/mcp-stdio.ts`, `Server` attached to `StdioServerTransport`). The handlers are byte-identical ŌĆö same Zod schemas, same MCP-facing extension with `llm_run_id`, same service-layer dispatch, same envelope. The stdio path has NO HTTP layer and NO JWT gate (see Auth row + `knowledge-graph.back.md` BR-01 stdio carve-out) ŌĆö the local OS process owner IS the trust boundary; the inviolable rule (`assertRunIsRunning` + 5-layer validation) is unchanged.
- **REST mirror:** the route is `POST /api/v1/ingest/llm-runs/:llmRunId/propose-{fragment,node,link,attribute}` ŌĆö the run id is in the URL path, the REST request schema does NOT carry `llm_run_id` in the body, the service verifies the same invariant via `assertRunIsRunning`.
- **Extraction orchestrator (in-process):** the orchestrator carries the `runContext` of the run it is driving and calls the services directly (no transport in the middle, no Anthropic-facing `llm_run_id` argument ŌĆö the orchestrator injects it server-side; the LLM is never asked for it).
**Description:** The single source of truth for the active-run invariant is the service layer (`assertRunIsRunning`). The MCP transport binds the run-id by tool argument (not by session); the REST mirror by URL path; the in-process orchestrator by injected `runContext`. The per-session model (`backend/src/modules/ingestion/mcp/session-factory.ts`) is **retired** ŌĆö the MCP endpoint is stateless single-shape (same pattern as `query` and `curation`, all three mounted via the shared SDK kernel `mountMcpEndpoint`, `backend/src/mcp/sdk-http-transport.ts`). The "ingest is MCP-only" constraint (┬¦14 / A28) remains **lifted** by v1.2.0 (documented deviation in CLAUDE.md).
**Error returned:** MCP envelope `{ ok: false, error.code: <namespaced> }` (P2.1: `VALIDATION_*` on Zod failure; `RESOURCE_NOT_FOUND` on unknown run id; `BUSINESS_RUN_NOT_RUNNING` on wrong status ŌĆö byte-identical to the REST wire code for the same condition) surfaced as MCP 2025-06-18 `content` / `isError: true` (HTTP `200` from the SDK kernel). REST propose-* mirror uses the same body shape but returns the real HTTP status (`404` for RESOURCE_NOT_FOUND, `409` for BUSINESS_RUN_NOT_RUNNING, `422` for VALIDATION_*) at the pre-handler layer. Auth/transport-level failures keep their normal HTTP error codes (`401` / `404` / `422`). **A `tool_call` row IS written on this path** ŌĆö the previous "pre-handler exception" of BR-23 no longer applies (see BR-23).

### BR-23 -- ToolCall always records the call (no transport exception)
**Related UC:** UC-08, UC-09, UC-10, UC-11
**Where to validate:** service ŌĆö the `propose_*` service function wraps the business transaction in a `try/finally`. ### BR-24 -- Tool schemas have a single source: the Zod DTOs (per-transport derivation)
**Related UC:** UC-08, UC-09, UC-10, UC-11
**Where to validate:** module-init code in `modules/ingestion/dto/`.   2. **MCP tool registration** ŌĆö registered with the shared SDK kernel `mountMcpEndpoint` (low-level `@modelcontextprotocol/sdk` `Server`). The MCP-facing schema is the business DTO **extended** with `llm_run_id: z.string().min(1)` (Option B ŌĆö arg-based run binding, see BR-21 / BR-28).
  3. **Anthropic tool-use call** (`client.messages.create({ tools: [...] })`) ŌĆö uses the business DTO unchanged (no `llm_run_id`; the orchestrator injects `runContext` server-side). **Error returned:** Not applicable (schema-derivation invariant; covered by unit test).

### BR-25 -- Entity resolution thresholds and decision branches
**Related UC:** UC-09
**Where to validate:** service (`entity-resolution.service.resolveOrCreateNode`), called by `propose-node.service` between the advisory lock acquisition (BR-20) and the INSERT (the previous code path's deferred extension point). Constants in `modules/ingestion/service/entity-resolution.service.ts`:
- `MATCH_STRONG = 0.85`
- `MATCH_FLOOR  = 0.55`

The procedure, in order:
1. **Exact match:** `SELECT node_id FROM node_alias WHERE alias_norm = norm($name) AND node_id IN (SELECT id FROM knowledge_node WHERE node_type_id = $ntid AND status = 'active')`. If any row, score is 1.0 ŌåÆ **reuse** that node; add any new aliases (`ON CONFLICT DO NOTHING` on `(node_id, alias_norm)`); resolution = `matched_existing`.
2. **Trigram candidates:** else
   ```sql
   SELECT na.node_id, MAX(similarity(na.alias_norm, norm($1))) AS sim
   FROM node_alias na
   JOIN knowledge_node kn ON kn.id = na.node_id
   WHERE kn.node_type_id = $2
     AND kn.status = 'active'
     AND na.alias_norm % norm($1)
   GROUP BY na.node_id
   ORDER BY sim DESC
   LIMIT 10;
   ```
   (uses `node_alias_norm_trgm_idx` GIN with `gin_trgm_ops`, filtered by FK index on `node_alias.node_id`).
3. **Decision (A12):**
   - **Strong unique:** exactly one candidate with `sim Ōēź MATCH_STRONG` AND no other candidate has `sim Ōēź MATCH_FLOOR` ŌåÆ **reuse** that node (add new aliases); resolution = `matched_existing`.
   - **Ambiguous:** any candidate has `sim Ōłł [MATCH_FLOOR, MATCH_STRONG)` OR two-or-more candidates have `sim Ōēź MATCH_STRONG` ŌåÆ create new node with `status = 'needs_review'`; for each ambiguous candidate (every row with `sim Ōēź MATCH_FLOOR`), INSERT one row into `entity_match_review (proposed_node_id, candidate_node_id, similarity)`. The new node and the review rows are inserted in the same transaction (atomic). Resolution = `needs_review`.
   - **Novel:** every candidate has `sim < MATCH_FLOOR` (including the empty case) ŌåÆ create new node with `status = 'active'`. Resolution = `created_new`.
4. **Alias attachment:** in all three branches above, every alias supplied by the LLM that is not yet on the resolved/created node is INSERTed into `node_alias` (`ON CONFLICT (node_id, alias_norm) DO NOTHING`). The canonical name is inserted as the first alias for new nodes.

The propose-node service then records a `tool_call` row with `validation_outcome = 'accepted'` for `matched_existing` / `created_new`, or `'needs_review'` for the ambiguous case. The Zod result schema (`ProposeNodeOutputSchema`) already carries `resolution: 'matched_existing' | 'created_new' | 'needs_review'`.

**Description:** Resolution is deterministic given the catalog and the live graph; thresholds are not configurable per call. The `entity_match_review` rows feed the `entity_match` curation queue (consumed by the existing `resolveEntityMatchService` / `performMerge` in `curation/service/merge.service.ts` ŌĆö no curation change required).
**Error returned:** Not a user-facing error path. Catalog miss on `node_type` is handled earlier (BR-14 ŌåÆ `BUSINESS_UNKNOWN_NODE_TYPE`, P2.1 namespaced).

### BR-26 -- Extraction orchestrator drives the LLM via a manual tool-use loop, one transaction per tool call
**Related UC:** UC-12 (new ŌĆö extraction trigger; declared in `ingestion.spec.md`)
**Where to validate:** service (`extraction.service.runExtraction`). Triggered by `POST /api/v1/ingest/llm-runs/:llmRunId/run`. The endpoint is synchronous (no worker, no queue) ŌĆö the HTTP request stays open for the duration of the run. Pre-checks: `llm_run.status = 'running'` (BR-21); a run that is already `completed` or `failed` is rejected with `409 BUSINESS_RUN_NOT_RUNNABLE`.

Algorithm (per ┬¦9.3, ┬¦9.4):
1. Load `llm_run` row, the `raw_information` row, and the chunks of that raw information (ordered by `chunk_index`).
2. Resolve the prompt module for `llm_run.prompt_version` via the prompt registry `modules/ingestion/prompts/index.ts` (`selectPromptModule`); an unregistered version throws `UnknownPromptVersionError`. The call runs inside the run-scoped try, so the throw flips the run to `failed` and surfaces `500 SYSTEM_INTERNAL_ERROR` (configuration error ŌĆö never silently substitute a prompt the run does not declare). Registry as of this revision: `v1` ŌåÆ `extraction.v1.ts`, `v2` ŌåÆ `extraction.v2.ts`. (Until the registry was introduced the orchestrator imported v1 statically, so `prompt_version` was recorded but did not drive the prompt; the registry closes that gap.)
3. Instantiate the Anthropic client (`new Anthropic({ apiKey: env.ANTHROPIC_API_KEY })`).
4. Derive the four tool defs from the four Zod schemas via `zod-to-json-schema` (BR-24).
5. For each chunk (in `chunk_index` order):
    a. Build `system = prompt.system(catalogSnapshot)`; build the user message:
       `prompt.user({ rawInformationMetadata, chunkText, prevTail, delimiterNonce })` ŌĆö `prevTail` is the last Ōēż 200 chars of the previous chunk's text, or empty for the first chunk. Chunk text is fenced by a per-run RANDOM nonce envelope `<<REMEMBER-DOCUMENT nonce=ŌĆ”>>` ŌĆ” `<<END-REMEMBER-DOCUMENT nonce=ŌĆ”>>` (┬¦13 anti-injection, hardened in P4.2 ŌĆö the document author cannot predict the nonce, so injected text cannot forge the closing marker to break out; `title`/`document_date` metadata are sanitized to a single line before interpolation). The nonce is one value per run, generated by the orchestrator. **`rawInformationMetadata` always carries `received_at` (ISO-8601 string of `raw_information.received_at`, set at intake to `now()` per UC-01), and this field is the canonical date-anchor consumed by `extraction.v3` to resolve relative date expressions (`"hoje"` / `"ontem"` / `"esta semana"` / similar pt-BR temporal deictics) in the chunk text when the document does not state an absolute date.** The orchestrator does NOT introduce a new argument or parameter for the anchor ŌĆö the existing `rawInformationMetadata.received_at` field is the anchor by contract; `extraction.v3`'s `user()` builder consumes it directly when composing the user prompt (see `prompts/extraction.v3.ts`). No new field on any DTO, no new column, no new MCP tool argument ŌĆö additive prompt-level use of an already-passed value. Earlier prompt versions (`extraction.v1` / `extraction.v2`) also receive `received_at` (legacy compatibility) but do not use it as a relative-date anchor; the behavioural change is internal to `extraction.v3` and any future prompt version that elects to consume the anchor.
    b. **Tool-use loop:** initialize `messages = [{ role: 'user', content: userBlocks }]`. Repeat:
        - Call `client.messages.stream({ model, system, tools, thinking: { type: 'adaptive' }, max_tokens, messages }).finalMessage()`.
        - **Append** the response's entire `content` array (preserves all `tool_use` blocks plus any `text` / `thinking` blocks) to `messages` as an `assistant` turn.
        - If `stop_reason === 'end_turn'`: chunk is done; break the loop.
        - If `stop_reason === 'pause_turn'`: continue the loop without modifying `messages` (Anthropic continues from the partial state).
        - If `stop_reason === 'refusal'`: log refusal, mark the chunk as a soft failure, break the loop (run continues with next chunk).
        - For each `tool_use` block in the response: call the matching service function ŌĆö `proposeFragmentService(client, args, runContext)` / `proposeNodeService(client, args, runContext)` / `proposeLinkService(client, args, runContext)` / `proposeAttributeService(client, args, runContext)` ŌĆö **each opening its own transaction (BR-19)**. Capture the verbatim envelope (`{ ok: true, result: { ..., validation_outcome } }` or `{ ok: false, error: {...} }`). **Per-tool-use uncaught-exception path (spec.md UC-12 alt 4b ŌĆö P2.1 namespaced binding):** if the service function throws an uncaught exception (DB outage, schema violation, `pg` client acquisition failure, internal bug ŌĆö NOT a layered-validation rejection, which is an in-band `{ ok: false, error }` result), the orchestrator catches it in the tool-use dispatch site (not the outer step-7 catch); the inner transaction has already rolled back (BR-19); the `try/finally` inside the service wrote a `tool_call` row with `validation_outcome = 'error'` (BR-23); the orchestrator synthesises a fallback envelope `{ ok: false, error: { code: 'SYSTEM_INTERNAL_ERROR', message: '<generic; raw err.message logged server-side only, never surfaced to the LLM>' } }` (P2.1 namespaced ŌĆö replaces the pre-P2.1 short `INTERNAL` that spec.md UC-12 alt 4b previously named; the wire code is byte-identical to the code REST already returns for uncaught exceptions per ┬¦6.2 of `ingestion.spec.md`). The loop then **continues to the next `tool_use` block in the same response** ŌĆö a single failing tool-use does NOT abort the chunk. Consecutive `'error'` outcomes trip the fatal-failure path only when they reach the burst threshold (spec.md UC-12 alt 4b ŌĆö Ōēź 3 in a row inside one chunk), at which point the orchestrator falls through to step 7 below. Build a `tool_result` block (`{ type: 'tool_result', tool_use_id, content: JSON.stringify(envelope), is_error: !envelope.ok }`) for each `tool_use` ŌĆö success, layered-validation rejection, and the synthesised `SYSTEM_INTERNAL_ERROR` envelope all use the same block shape. Append a single user turn with all `tool_result` blocks; restart the loop.
6. After all chunks complete: close the run ŌĆö `UPDATE llm_run SET status = 'completed', finished_at = now() WHERE id = $1` (single transaction). Emit a `pino` info log with the eight `validation_outcome` counts of the run (BR-12, BR-26 closing path).
7. **Fatal failure path:** if the orchestrator catches an unhandled exception at any point (network, model 5xx, internal bug ŌĆö NOT layered-validation rejection, which is an in-band result), it issues `UPDATE llm_run SET status = 'failed', finished_at = now() WHERE id = $1` in a fresh transaction, then re-throws. The HTTP response is `500 SYSTEM_INTERNAL_ERROR`. The owner re-runs by invoking `retryLlmRun` (BR-10) followed by `runLlmExtraction` again.

The orchestrator **never** wraps multiple `propose_*` calls in one transaction (┬¦9.4): a single fragment / node / link / attribute persists or rolls back independently. The `tool_call` rows are the audit trail; the run summary (BR-12) is the human-facing aggregation.

**Description:** Synchronous orchestration is acceptable at the current scale (hundreds of documents, ┬¦16); a queue / worker offload is a future evolution. The orchestrator is the single in-process LLM caller of the BFF; no other code path calls Anthropic.
**Error returned:** `409 BUSINESS_RUN_NOT_RUNNABLE` if the run is not `running` at entry. `500 SYSTEM_INTERNAL_ERROR` on orchestrator fatal failure (after the run has been moved to `failed`). `200 OK` with a `RunExtractionResponse` body on success.

### BR-27 -- Graph consolidation on `propose_link` / `propose_attribute`
**Related UC:** UC-10, UC-11
**Where to validate:** service (`graph-consolidation.service`) invoked by `propose-link.service` and `propose-attribute.service` after the 5-layer validation (BR-13..BR-18) has passed and the new assertion has been pre-built (target `value`, `valid_from`, `valid_from_basis`, `change_hint`, etc.).

The consolidator implements ┬¦6.5 in this exact order, inside the same transaction as the `propose_*` call (BR-19):

1. **Scope identification:**
   - For `propose_link`: scope = `(source_node_id, link_type_id, target_node_id)`. Functional flag = `link_type.allows_multiple_current` (false ŌåÆ functional).
   - For `propose_attribute`: scope = `(node_id, attribute_key_id)`. Functional flag = `attribute_key.allows_multiple_current`.
2. **Vigent lookup (A11):**
   ```sql
   SELECT * FROM knowledge_link
   WHERE source_node_id = $1 AND link_type_id = $2 AND target_node_id = $3
     AND valid_to IS NULL AND superseded_at IS NULL
   FOR UPDATE;
   ```
   (mirror for `node_attribute`, scoped by `(node_id, attribute_key_id)`). Lock is held until commit/rollback. This is what closes the race that the partial dup-guard index would otherwise turn into an SQLSTATE `23505`.
3. **Decision tree (┬¦6.5):**
   - **Re-affirmation (consolidation):** vigent row exists; same value (target node for link, parsed value for attribute); `change_hint = 'none'`. ŌåÆ Do **not** insert a new row. Instead, INSERT one `provenance` row per cited fragment, attached to the existing assertion's id (┬¦18). `tool_call.validation_outcome = 'consolidated'`. Response: `{ ok: true, result: { outcome: 'consolidated', link_id | attribute_id: <existing id> } }`. The `valid_from` of the new proposal is **not** part of the re-affirmation identity in any of the three sub-cases below ŌĆö `valid_from` is metadata of the witnessing document, not of the asserted fact (┬¦6.5 evidence-basis precedence: `stated > document > received`). The vigent row's `valid_from` is preserved as-is on consolidation; this revision does NOT yet promote an existing `received`-basis `valid_from` to a `stated`-basis one when a later document asserts the same fact with a stronger basis ŌĆö that promotion is a separate evolution under ┬¦6.5 and is **out of scope** for this fix (see ┬¦8). The three re-affirmation sub-cases are:
     - **Sub-case (i) ŌĆö multi-current link / multi-valued attribute (`allows_multiple_current = true`):** the dup-guard partial UNIQUE is scoped by `(source, link_type, target)` (or `(node, attribute_key, value)`) **without** `valid_from`; therefore one and only one vigent row per scope is permitted by the schema, and re-affirmation MUST consolidate regardless of whether the new proposal's `valid_from` matches the vigent row's `valid_from`. Concretely: when `link_type.allows_multiple_current = true` (or `attribute_key.allows_multiple_current = true`), the consolidator takes this branch whenever the vigent row's target/value matches the new proposal's target/value and `change_hint = 'none'` ŌĆö even when `vigent.valid_from <> new.valid_from`. This is the canonical case for the cross-document re-affirmation of an LLM-extracted fact whose `valid_from` comes from the per-document `received` fallback (FR-001 in `temporal.ts`): the second document is recorded on a different day, so its `received` basis yields a different date, but the fact is the same and ┬¦18 requires consolidation.
     - **Sub-case (ii) ŌĆö functional link / single-valued attribute (`allows_multiple_current = false`) with identical `valid_from`:** the consolidator takes this branch when `vigent.target_node_id = new.target_node_id` (or `vigent.value = new.value` for attributes) AND `vigent.valid_from = new.valid_from` AND `change_hint = 'none'`. This preserves the previous v1.2.1 behaviour for functional scopes; the `valid_from` equality is required here because for functional types a divergent `valid_from` together with a different value carries semantic meaning (succession ŌĆö sub-case (iii) and the succession branch below).
     - **Sub-case (iii) ŌĆö functional link / single-valued attribute with same target/value but divergent `valid_from`, `change_hint = 'none'`, no textual succession or correction signal:** treat as re-affirmation (same fact, the divergent date is from a different basis of evidence). Consolidate as in sub-case (ii); the vigent row's `valid_from` is preserved. Rationale: a same-value functional assertion with a different `valid_from` and no succession signal is, by ┬¦6.5, the same fact re-asserted ŌĆö `valid_from` of the new proposal is then informational only and does NOT trigger succession (succession requires a different target/value AND a textual signal ŌĆö see the succession branch below).
   - **Succession (functional types only):** vigent row exists; functional scope; new value differs from old; the supporting fragment text contains a textual succession signal AND/OR `change_hint = 'succession'`. ŌåÆ UPDATE the vigent row to set `valid_to = $newValidFrom` (or `now()::date` if the new row has no `valid_from`), `superseded_at = now()`, `status = 'superseded'`. **Intra-day collapse guard:** the validity axis is `date` (day-granular, ┬¦5.1), so `valid_to = $newValidFrom` is applied ONLY when it is strictly after the vigent row's `valid_from`. When `vigent.valid_from >= $newValidFrom` (a same-day ŌĆö or earlier-dated ŌĆö change; typically both proposals resolving `valid_from` via the `received` fallback on the same day), setting `valid_to = $newValidFrom` would yield a degenerate `[D, D)` interval and violate the strict `valid_from < valid_to` CHECK (`knowledge_link_interval_ck` / `node_attribute_interval_ck`; also enforced in `temporal.ts`). In that case the vigent row is closed on the **transaction axis only** (`superseded_at = now()`, `status = 'superseded'`, `valid_to` left untouched) ŌĆö identical to the Correction sub-case ŌĆö so the new version becomes current via the `supersedes_*` lineage + the `timestamptz` transaction axis. The sub-day validity boundary is intentionally dropped: day-granular validity cannot encode two effective values on the same calendar day (the literal ┬¦6.5-A wording `valid_to = data_da_mudan├¦a` is unsatisfiable here). Implemented by `closeVigentForSuccession` in `graph-consolidation.service.ts` (a SQL `CASE`, evaluated against the row's own `valid_from` and the DB clock). INSERT the new row with `supersedes_link_id` / `supersedes_attribute_id` set to the old row's id. INSERT `provenance` rows attached to the new id. `tool_call.validation_outcome = 'superseded_previous'`. Response: `{ ok: true, result: { outcome: 'superseded_previous', link_id | attribute_id: <new>, superseded_id: <old> } }`.
   - **Correction (errata):** vigent row exists; `change_hint = 'correction'` with the textual errata signal verified by BR-16; same period. ŌåÆ Apply the ┬¦6.5 correction flow: close the vigent row (`status = 'corrected'`, `superseded_at = now()`, do NOT change `valid_from`/`valid_to` of the old row), INSERT a new row with the corrected value and `supersedes_*` set; `provenance` attached to the new id. `tool_call.validation_outcome = 'accepted'` (per spec.md BR-25: correction is an accepted write; the audit trail lives in the supersedes-* chain and in `tool_call.result.reason`).
   - **Conflict (dispute):** vigent row exists; same period (overlapping `[valid_from, valid_to)`); different value; no succession signal; no correction errata. ŌåÆ Two outcomes per ┬¦6.5 flow C: UPDATE the vigent row to `status = 'disputed'`; INSERT the new row also with `status = 'disputed'`; both rows now coexist as `disputed`; `provenance` attached to the new id. `tool_call.validation_outcome = 'disputed'`. Response: `{ ok: true, result: { outcome: 'disputed', link_id | attribute_id: <new>, conflicting_id: <old> } }`.
   - **New assertion (no vigent row, or non-functional scope with no overlap):** INSERT a new row with `status` derived from BR-17 (`active` / `uncertain` / `disputed`). INSERT `provenance` rows attached to the new id. `tool_call.validation_outcome = 'accepted'`. Response: `{ ok: true, result: { outcome: 'accepted', link_id | attribute_id: <new> } }`.

The consolidator never executes a plain INSERT against a vigent scope that already has a row ŌĆö the `FOR UPDATE` lookup is mandatory. The pre-existing partial dup-guard indexes (`knowledge_link_current_dup_guard`, `node_attribute_current_dup_guard`) remain in place as a defence-in-depth safety net.

**Dup-guard catch path (`23505` SQLSTATE on the partial dup-guard).** When the consolidator's `INSERT` raises SQLSTATE `23505` on either of those two constraint names, the cause is one of two things, which the service MUST distinguish: (1) a **deterministic conflict** ŌĆö the consolidator's decision tree above missed the vigent row (e.g. the vigent lookup ran before another transaction commit raced the lock; or a sub-case branch was not taken because of a stale field); the correct response is to **re-run the lookup-and-decide step** (`SELECT ... FOR UPDATE` plus the decision tree above), which will now observe the freshly-committed vigent row and route to consolidation (sub-case (i)/(ii)/(iii)) or to succession/correction/dispute; (2) a **real concurrent contention** ŌĆö another transaction is sitting in `FOR UPDATE` on the same scope and only releases its lock after our retry. The service implements **one** retry of the lookup-and-decide step after a `23505` catch. If the second attempt also raises `23505`, this is a real-time race that cannot be resolved deterministically by retrying ŌĆö the call is rejected with `VALIDATION_INVALID_FORMAT` (P2.1 namespaced ŌĆö the concurrent contention manifests as a structural-layer rejection surface) and the error message MUST identify it as such (e.g. `"graph consolidation: real concurrent contention on <scope>; one of the racing transactions must be retried by the caller"`). The previous catch-all message (`"hit dup-guard twice; concurrent contention not resolvable"`) is forbidden ŌĆö it conflates a deterministic miss of sub-case (i) (which the BR-27 revision above now resolves on the retry) with a real concurrent race (which the retry cannot resolve). On the retried path, the consolidator MUST take the re-affirmation branch above whenever the vigent row matches target/value and `change_hint = 'none'` ŌĆö including the multi-current sub-case (i) with divergent `valid_from`. If the consolidator's decision tree is correctly implemented per the sub-cases above, the second dup-guard fire must be vanishingly rare (real concurrent writes to the same `(source, link_type, target)` within the few-ms window between the `FOR UPDATE` release and our INSERT ŌĆö possible but improbable at v1's single-owner scale).

**Description:** Re-running an extraction over the same document re-affirms the existing graph rather than failing on dup-guard. Re-mentioning an existing entity in a second chunk reuses the existing node (via BR-25) and re-asserts its links by consolidation (via BR-27).
**Error returned:** Not a user-facing error path ŌĆö every branch returns `{ ok: true, result }`. Programming-error fall-through (e.g. consolidator skipped) ŌåÆ SQLSTATE `23505` on the partial dup-guard ŌåÆ `500 SYSTEM_INTERNAL_ERROR`.

When omitted, `model` defaults to the server's recommended model (`claude-opus-4-8`, ┬¦1 Anthropic-client row) and `prompt_version` to `DEFAULT_PROMPT_VERSION` (`'v2'`, BR-26).
**Description:** Exposes the UC-01 + UC-12 composition as a single MCP tool so an external MCP client (e.g. Claude Desktop) can ingest a whole document without managing run lifecycle or chunk offsets. The extraction LLM is the **server's** (`ANTHROPIC_API_KEY`, BR-29) ŌĆö the client only supplies the content; the inviolable rule (the LLM never touches the DB directly, ┬¦2) holds, since every write still flows through the validated propose-* path the orchestrator calls. Synchronous and LLM-bound (same latency profile as UC-12 ŌĆö the HTTP connection is held open for the whole run; minutes for long documents). **MCP-facing only**: there is no REST mirror, because the REST surface already exposes UC-01 (`POST /raw-information`) and UC-12 (`POST /llm-runs/:id/run`) as separate endpoints. **Reachable over BOTH MCP transports (Emenda v7.5):** the HTTP transport (`POST /api/v1/mcp/ingest`) AND the local stdio transport (`backend/src/mcp-stdio.ts`) ŌĆö same handler, same `Server` registry, same service-layer call (`ingestRawInformation` + `runLlmExtraction`). The stdio path lets an external local MCP client (e.g. Claude Desktop) drive the full UC-01 + UC-12 composition without running the Neon Auth OAuth flow: the local OS process owner IS the trust boundary. DI seams (`ingestRaw` / `runExtraction` / `readRunStatus`) on the handler let unit tests exercise the branch + error-mapping logic without a database (same convention as `anthropicFactory` / `now`).
**Known limitation (recovery):** because the idempotent path never re-extracts, a content whose FIRST run failed cannot be recovered through this tool (re-sending returns `already_ingested` with the failed `run_status`). The owner recovers via the REST surface (`POST /llm-runs/:id/retry` then `/run`, UC-06 + UC-12); a dedicated MCP retry/re-run tool is deferred (would be a new ┬¦14 tool ŌĆö out of scope here).
**Error returned:** Zod failure ŌåÆ `{ ok: false, error.code: <VALIDATION_REQUIRED_FIELD | VALIDATION_INVALID_FORMAT | VALIDATION_OUT_OF_RANGE> }` (P2.1 Zod-discriminated). Intake failure (step 2) ŌåÆ `SYSTEM_SERVICE_UNAVAILABLE` when the database is unavailable (`isPgUnavailable`), else `SYSTEM_INTERNAL_ERROR` (P2.1 namespaced) with a generic message (the raw `err.message` is logged server-side only, never returned). Provider-fatal during extraction ŌåÆ `{ ok: false, error.code: "SYSTEM_LLM_PROVIDER_UNAVAILABLE", details: { llm_run_id, raw_information_id, partial_run } }` (`LlmProviderFatalError`, BR-26); extraction-fatal ŌåÆ `{ ok: false, error.code: "SYSTEM_INTERNAL_ERROR", details: { ŌĆ” } }` (`ExtractionFatalError`); any unexpected error ŌåÆ `SYSTEM_INTERNAL_ERROR` (P2.1 namespaced). On the extraction paths the orchestrator has already closed the run `failed` (BR-26). All surfaced as MCP 2025-06-18 `content` / `isError: true` (HTTP `200` from the SDK kernel). **No new error code** ŌĆö every code is already registered in `docs/specs/_global/error-codes.md`.

### BR-31 -- Read-only operational tools on the `ingest` toolset (`health`, `get_ingestion_status`, `list_recent_ingestions`)
**Related UC:** UC-04 (run status), plus operational liveness.
**Where to validate:** registered in `mcp/ingest-toolset.ts` on the `ingest` toolset and advertised by `POST /api/v1/mcp/ingest` (whitelist composed in `app.ts`). An external MCP client can now (a) confirm the BFF is up before ingesting (`health`), and (b) after a timeout, recover the run id by preview/source_type (`list_recent_ingestions`) and read its outcome (`get_ingestion_status`) **instead of blindly re-sending** (a re-send of reformatted content has a different `content_hash` ŌåÆ a DUPLICATE `RawInformation`, since idempotency keys on exact bytes ŌĆö BR-08/BR-09). This is the read-only half of the timeout mitigation; making `ingest_document` itself asynchronous (return the run id immediately + poll) is a **separate, contract-changing** change and is **out of scope here** (it would amend BR-30 + ┬¦14).
**Error returned:** `VALIDATION_INVALID_FORMAT` (Zod) ┬Ę `RESOURCE_NOT_FOUND` (unknown run) ┬Ę `SYSTEM_SERVICE_UNAVAILABLE` (db down) ┬Ę `SYSTEM_INTERNAL_ERROR` (unexpected). **No new error code.**

### BR-32 -- `start_async_ingestion` -- **WITHDRAWN in v1.4.0**
**Related UC:** Former UC-13 (async extraction). UC-13 is **retired** alongside this BR ŌĆö see `ingestion.spec.md` changelog.
**Status:** Withdrawn 2026-06-25. The tool `start_async_ingestion` is no longer registered on the `ingest` toolset, no longer advertised over either MCP transport, and no longer reachable from the chat module. The sole consumer (the chat agentic loop) now uses the deterministic synchronous `ingest_directed` tool (BR-34) instead.
**Rationale for retirement:**
- The original design (`v1.2.9`) added an LLM round-trip on the **server** (the orchestrator extracted the synthesised chat content with an Anthropic call) plus a detached-promise lifecycle that could leave runs orphaned on BFF restart. Both costs only existed to compensate for the chat agent not knowing exactly **what** to ingest.
- The `v1.4.0` redesign moves the structuring responsibility back to the chat LLM (which already understands the turn context): the chat agent emits a directed payload, the server runs it deterministically, no second LLM call is needed, and the response is bounded by intake + N propose-* transactions (sub-second to low seconds for typical chat payloads).
- The legacy contract (`mcp.getTool('ingest', 'start_async_ingestion')`) is removed; callers that still reference it MUST migrate to `ingest_directed` (BR-34).
**Recovery path for existing orphaned runs.** Runs that were created by `start_async_ingestion` before v1.4.0 and remain in `status = 'running'` are recovered exactly as any other interrupted run: the Owner reads `llm_run.status` via `get_ingestion_status` (BR-31), and if stale, drives `retryLlmRun` (BR-10) followed by `runLlmExtraction` (UC-12). No data is lost ŌĆö every `RawInformation` / `RawChunk` / `LLMRun` / `tool_call` row already written by an async run is preserved.
**Description:** Historical only. The number BR-32 is reserved (not reused) to preserve traceability with the v1.2.9..v1.3.0 changelog entries and with the bug reports / DEC entries that referenced it.
**Error returned:** Not applicable ŌĆö the tool no longer exists. A residual MCP `tools/call` against `start_async_ingestion` returns the SDK kernel's standard `Method not found` shape (the tool is not registered), not a business envelope.

### BR-33 -- Extraction orchestrator collects affected-node outcomes; surfaced on `LLMRun` completion

**Related UC:** UC-04 (LLMRun read), UC-09 (entity resolution), UC-10 (link consolidation), UC-11 (attribute consolidation), UC-12 (extraction), UC-14 (directed ingestion, declared in `ingestion.spec.md` v1.4.0 alongside BR-34; the former UC-13 async-extraction composition is **withdrawn**).

**Motivation.** Today the run summary (BR-12) carries only counts (`accepted`, `consolidated`, `superseded_previous`, `needs_review`, `uncertain`, `disputed`, `rejected`, `error`). A chat agent that wants to surface "what was just ingested" after a directed-ingestion (`ingest_directed`, BR-34) ŌåÆ `get_ingestion_status` (BR-31) cycle (the prior async path `start_async_ingestion` of v1.2.9 is **withdrawn** in v1.4.0 ŌĆö see BR-32) has no entity-grained surface and is forced to guess a full-text search (which is `AND`-semantics; multi-name queries return 0). Surfacing the ids+canonical_name+node_type of every `KnowledgeNode` the run touched eliminates the guessing step: the caller jumps directly to `get_node` / `traverse`.

**Where to validate:** service (`extraction.service.runExtraction`) and `llm-run.service.getLlmRunById` / `findRecentIngestions` (read paths).

**Collection (orchestrator side, BR-26 step 5b).** Whenever the orchestrator dispatches a `tool_use` block for `propose_node`, `propose_link`, or `propose_attribute` and the matching service returns `{ ok: true, result }` with one of the affected-node outcomes below, the orchestrator records the affected node id(s) on a per-run in-memory `Map<uuid, AffectedNodeEntry>` (de-duplicated by node id; later observations of the same id are no-ops):
- **`propose_node`:**
  - `result.outcome === 'created_new'` (BR-25): the newly created `node_id` (status active or `needs_review`).
  - `result.outcome === 'matched_existing'` (BR-25): the reused `node_id`.
  - `result.outcome === 'needs_review'` (BR-25): the newly created `node_id` (the candidate ids are NOT included ŌĆö they are not the run's output, they are inputs to a curation queue and are observable via `entity_match_review`).
- **`propose_link`:** the `source_node_id` and `target_node_id` of the new / consolidated / superseded / disputed link, for every BR-27 outcome that returns `{ ok: true }` (`accepted`, `consolidated`, `superseded_previous`, `disputed`; the correction sub-case also surfaces via `validation_outcome = 'accepted'`).
- **`propose_attribute`:** the `node_id` whose attribute was touched, for every BR-27 outcome that returns `{ ok: true }`.

`rejected` outcomes (BR-13..BR-18) and `error` outcomes (uncaught exceptions) do NOT contribute ŌĆö they did not touch the graph.

**De-duplication and ordering.** The Map is keyed by `node_id`; first-write-wins on the entry (the `canonical_name` / `node_type` resolved at first observation is the stable label for the run). Iteration order on the final list is the insertion order (deterministic by per-chunk tool-use order); no sort by id or name (callers that need an ordering do their own sort, e.g. alphabetical by `canonical_name`).

**Resolution to the final shape (`{ id, canonical_name, node_type }`).** After all chunks complete (BR-26 step 6) and before emitting the run-completion log, the orchestrator issues ONE batched lookup against `knowledge_node` JOIN `node_type` for the collected ids:
```sql
SELECT kn.id, kn.canonical_name, nt.name AS node_type
FROM knowledge_node kn
JOIN node_type nt ON nt.id = kn.node_type_id
WHERE kn.id = ANY($1::uuid[]);
```
Result rows that match active or `needs_review` nodes are included verbatim; rows whose `knowledge_node.status = 'merged_into'` (i.e. a candidate node merged by curation between the tool call and the run-completion lookup) are **resolved transparently** to the surviving node (`knowledge_node.merged_into_node_id`) by re-querying for the survivor; if the survivor itself has been merged the resolver follows the chain once (path compression on write ŌĆö CLAUDE.md "Known Gotchas" ŌĆö normally keeps this at depth 1).

**Persistence.** The list is persisted **on the `LLMRun` row** ŌĆö a new nullable column `llm_run.affected_nodes_snapshot jsonb` (added by a future additive migration; see Out of Scope below for the v1.3.0 in-memory deferral). Until the migration lands, the list lives in an in-process cache keyed by `llm_run_id` with a bounded LRU (size 256, eviction-tolerant: a cache miss on a completed run falls back to a **derived** lookup over `tool_call.result` rows for that run, re-running the de-dup + batched lookup above). Both surfaces ŌĆö cached and derived ŌĆö produce byte-equivalent payloads; the column write is a write-through that turns the cache into a soft pre-warm. This BR specifies the **contract** (the field on `get_ingestion_status` and the synchronous BR-30 return); the storage choice is implementation-internal and may evolve without changing the contract.

**Read paths.**
- `get_ingestion_status` (BR-31) attaches `result.affected_nodes` ONLY when `llm_run.status === 'completed'`; absent or `null` otherwise (a `running` or `failed` run does not surface a partial list ŌĆö partial extraction state is observable via `tool_call` rows for forensics, not via this field).
- `ingest_document` (BR-30) synchronous return embeds the same list under `run.affected_nodes` on the success path (status implicitly `completed`); on the `already_ingested` / `not-yet-completed` branches the list is absent (caller polls `get_ingestion_status`).
- `ingest_directed` (BR-34) synchronous return embeds the list under `result.run.affected_nodes` on the `ingested` branch (same shape as `ingest_document` BR-30 ŌĆö the field lives under `run`, not at the top of `result`); the directed orchestrator finishes inline and the run is always `completed` on a successful intake (BR-34 step 5), so the field is populated on every successful return. **There is no idempotent branch on the directed path** (BR-34 step 2 ŌĆö the per-call nonce makes every `content_hash` unique): the read-path of this field on `ingest_directed` is unconditional on intake success. The legacy `start_async_ingestion` row of this list (v1.3.0) is **withdrawn** alongside BR-32.

**Type.** Optional field on the run result envelope. TypeScript shape:
```ts
type AffectedNode = { id: string; canonical_name: string; node_type: string };
// `result.affected_nodes` is `Array<AffectedNode> | undefined` (never `null` on the wire ŌĆö serializers omit the key entirely when undefined).
```
The Zod schema for the response DTO declares it as `.optional()`. Empty array (`[]`) is a valid completed-run payload (a run can complete with only `rejected` outcomes); it is NOT the same as absent (absent means the run is not in a state that supports the field).

**Additive change.** This BR is additive on every transport:
- Existing consumers that do not read `affected_nodes` are unaffected (the field is `optional` on the Zod schema; JSON serializers omit it when undefined).
- The MCP / REST envelope shape is unchanged; the field lives inside `result`, which is already an open object.
- No new error code, no new HTTP status, no new MCP tool, no new endpoint, no DDL in v1.3.0 (the `llm_run.affected_nodes_snapshot` column is deferred to a future minor revision; see Out of Scope).

**Cache invalidation.** A subsequent merge of one of the affected nodes (via the curation `performMerge`, separate from this domain) does NOT retroactively rewrite the snapshot of a completed run ŌĆö the field is the snapshot of the run at completion. A consumer that needs the up-to-date survivor calls `get_node(id)` (the survivor is reachable via `merged_into_node_id`); this is the same contract the chat agent already follows for any node id.

**Description:** The run's affected-node list is a derived projection of the orchestrator's per-tool-use outcomes ŌĆö the same data the `tool_call` rows already carry, surfaced as a flat ergonomic list on the `LLMRun` read path. The chat agentic loop (consumer; `chat.back.md` BR-18 v3 ontology-aware prompt) is the canonical first reader; an external MCP client (e.g. Claude Desktop) is the second.

**Error returned:** Not a user-facing error path. A failure during the batched `knowledge_node` lookup at run-completion (e.g. transient DB outage) is logged at WARN and the field is omitted from the run-completion log ŌĆö the run is still recorded as `completed` and the caller re-derives the list on the next `get_ingestion_status` call (which re-runs the same lookup). The contract is best-effort, not transactional: surfacing the field is a read-time projection, not a write-time invariant.

### BR-34 -- `ingest_directed` -- one-shot, deterministic directed ingestion tool (MCP)

**Related UC:** UC-01, UC-08, UC-09, UC-10, UC-11, and the new UC-14 (declared in `ingestion.spec.md` v1.4.0 ŌĆö directed-ingestion composition; deterministic sibling of the legacy UC-13).

**Where to validate:** MCP handler `modules/ingestion/mcp/directed-ingest.handler.ts`, registered on the `ingest` toolset (`mcp/ingest-toolset.ts`) and advertised by `POST /api/v1/mcp/ingest` (whitelist composed in `app.ts`). Reachable over BOTH MCP transports (HTTP and local stdio) ŌĆö same `Server` registry, same handler, same envelope, mirroring BR-30 / former BR-32 (Emenda v7.5). Backed by a new service module `modules/ingestion/service/directed-ingestion.service.ts` that implements the deterministic orchestrator described below.

**Tool schema.** Zod schema `IngestDirectedMcpInputSchema` in `mcp/mcp-schemas.ts`; description in `dto/index.ts#IngestToolDescriptions.ingest_directed`:

```
ingest_directed {
  fragments:    [{ ref: string, text: string(1..1000) }]  // 1..N
  nodes:        [{ ref: string, node_type: string, name: string(1..500), node_id?: uuid, aliases?: string[] }]  // 1..N
  attributes?:  [{ node_ref: string, key: string, value: string|number|boolean, evidence_ref: string, valid_from?: date, valid_from_basis?: 'stated' | 'document' }]
  links?:       [{ source_ref: string, target_ref: string, link_type: string, evidence_ref: string, valid_from?: date, valid_from_basis?: 'stated' | 'document' }]
  source_label?: string  // free-form caller-supplied tag used only inside the synthesised content (audit aid; not parsed)
}
```

Notes on the schema:
- `ref` strings are **local to the call** (their scope is this one tool invocation; they are never persisted or returned). The handler builds in-memory maps `ref -> fragment_id` / `ref -> node_id` and resolves every cross-reference at dispatch time.
- `confidence` is deliberately **absent** from every item ŌĆö the server forces `confidence = 1.0` on every dispatched `propose_*` call (see Step 4 below). Callers cannot lower confidence on this tool; the directed payload is a **stated fact** by construction.
- `valid_from_basis` is restricted to `'stated' | 'document'` per the public `ValidFromBasis` enum (the `'received'` fallback is server-internal, never accepted from callers ŌĆö `ingestion.spec.md` ┬¦4 BR-16; `openapi.yaml#components/schemas/ValidFromBasis`).
- `node_id` on a node item is a **pin** (UUID): when supplied, the handler skips entity resolution and uses the supplied id directly ŌĆö the chat agent uses this when it already retrieved the node via `query`-toolset reads and wants to re-affirm against it without risking a trigram drift to a different node. When omitted, the handler runs the standard `propose_node` resolution (BR-25 ŌĆö exact alias, trigram candidate, three-way decision under advisory lock BR-20).

**Determinism (the core distinction from BR-30).** `ingest_directed` does **NOT** call Anthropic. The `LLMRun` row is opened with the sentinels `model = 'directed'` and `prompt_version = 'directed-v1'`; the orchestrator never invokes `selectPromptModule` / `runExtraction` for this run. The Anthropic API key is irrelevant on this path; the LLM cost of this tool is zero on the server. The five-layer validation pipeline of BR-13..BR-18 plus the entity-resolution thresholds of BR-25 plus the graph-consolidation logic of BR-27 plus the audit invariant of BR-23 remain unchanged ŌĆö every accepted item still flows through the same `propose_*` service functions that the LLM-driven extraction (BR-26) and the REST mirrors (BR-28) use. The only thing absent on this path is the Anthropic round-trip; the inviolable rule (┬¦2 ŌĆö every write flows through a validated `propose_*` handler) is preserved by construction.

**Flow.**
1. **Zod-parse** the payload (envelope `{ ok:false, error.code: <VALIDATION_REQUIRED_FIELD | VALIDATION_INVALID_FORMAT | VALIDATION_OUT_OF_RANGE> }` ŌĆö P2.1 Zod-discriminated ŌĆö on failure; no run row is opened, no `tool_call` row is due ŌĆö there is nothing to audit against yet).
2. **Synthesise raw content + open the run (1 TX, BR-19):** concatenate the `fragments[].text` (one fragment per line, prefixed with `[ref]`) with a trailing **timestamp + nonce** line (`-- directed_at=ISO8601 nonce=base64url(16 bytes)`). This guarantees a unique `content_hash` per directed call ŌĆö re-running the tool with identical fragments yields a different `RawInformation` (and therefore a different `LLMRun`), so there is **no `noop_existing` branch on the directed path** (decision 3 of the approved plan). `ingestRawInformation` (`ingestion.service.ts`) persists the `RawInformation` (`source_type = 'chat'`, `metadata.directed = true`, `metadata.source_label` carrying the caller-supplied tag when present, plus `metadata.conversation_id` / `metadata.message_id` when supplied by the chat route ŌĆö see step 2bis below; both are non-PII routing identifiers ŌĆö the chat module's stable UUIDs ŌĆö and are safe to carry in the persisted `metadata` jsonb), the chunk(s), and the `LLMRun` (`status = 'running'`, `model = 'directed'`, `prompt_version = 'directed-v1'`). Intake errors are caught and mapped to a clean envelope (`SYSTEM_SERVICE_UNAVAILABLE` when `isPgUnavailable`, else `SYSTEM_INTERNAL_ERROR` ŌĆö P2.1 namespaced ŌĆö with a generic message) ŌĆö never re-thrown (same pattern as BR-30 step 2; prevents the SDK kernel from leaking `err.message`).

   **Step 2bis ŌĆö server-side verbatim capture of the chat turn (v1.5.0).** The directed-ingestion handler `modules/ingestion/mcp/directed-ingest.handler.ts` accepts an **optional second argument** `invocation_context?: { source_excerpt?: string; conversation_id?: string; message_id?: string }` alongside the Zod-validated tool input (the first argument). The handler forwards `source_excerpt` to `runDirectedIngestion(deps)` as `deps.sourceExcerpt`; the directed-ingestion service in turn passes it to `ingestRawInformation` as the new `original_input` field of `IngestRawInformationRequest`. The repository (`ingestion.repository.ts`) extends `RawInformationRow` with `original_input: string | null` and includes the column in `insertRawInformation`'s INSERT column-list and `RETURNING` projection. **`original_input` is server-side and deterministic; it is NEVER an argument of the MCP tool schema** ŌĆö the LLM is never asked for the verbatim turn (an LLM-supplied verbatim would defeat the purpose: the LLM is the layer that may have silently corrected typos, paraphrased, or otherwise mutated the user's input; the ┬¦1 anti-silent-discard principle requires capture *before* the LLM acts on the input). The capture mechanism is the **chat dispatch path only** (transport-neutral but populated only on the chat caller side): `conversations.routes.ts` threads `body.content` (verbatim) as `ctx.input.current_user_turn`; `chat-agent.service.ts`'s generic dispatcher (`raceToolHandler`) gains a third parameter `invocation_context` that is forwarded to every tool handler (no per-tool branch ŌĆö only `directed-ingest.handler.ts` reads `source_excerpt`, every other handler ignores it); the chat agent populates `invocation_context = { source_excerpt: ctx.input.current_user_turn, conversation_id: ctx.input.conversation_id, message_id: ctx.input.message_id }` immediately before dispatch. All other callers ŌĆö the REST `propose_*` mirrors (BR-28), `ingest_document` (BR-30), direct MCP calls to `ingest_directed` from an external client (Claude Desktop, `mcp-remote`), and the in-process LLM extraction orchestrator (BR-26) ŌĆö invoke the handler **without** an `invocation_context` argument, so `original_input` lands as `NULL` on the inserted `raw_information` row. The `conversation_id` / `message_id` ride alongside as the `metadata.conversation_id` / `metadata.message_id` pointer (see step 2 above) so the row remains forensically linkable to the originating `chat_message` even though `chat_message` lives outside ┬¦11 (the `original_input` column is the ┬¦11-covered surface; the pointer is a non-PII routing aid). Idempotency is unaffected (BR-01 / BR-08 unchanged): `content_hash` is computed over `content` only, and `content` continues to be the synthesised fragments-plus-nonce block ŌĆö two `ingest_directed` calls with identical fragments still hash differently because of the nonce in `content`, regardless of `original_input`. Re-affirmation semantics (the existing graph-consolidation path of BR-27) are likewise unchanged ŌĆö the per-item `propose_*` calls still hit `consolidated` / `accepted` / `superseded_previous` / `disputed` outcomes against the existing `knowledge_link` / `node_attribute` rows; `original_input` is a `raw_information`-row property, not a graph-layer property, so consolidation does not read or write it. Compliance redaction (┬¦11): the `original_input` column is covered by the same `compliance_delete` flow that redacts `raw_information.content` ŌĆö see `compliance-audit.back.md` (`tombstoneRawInformation` SQL UPDATE clause). The redaction is conditional on non-null (`original_input = CASE WHEN original_input IS NULL THEN NULL ELSE '[REDACTED]' END`) ŌĆö null rows stay null, non-null rows become `'[REDACTED]'`; the chat pointer in `metadata` is also redacted by the same compliance flow (see compliance back-spec BR for the exact metadata redaction shape).
3. **Dependency-ordered dispatch.** With the run open, the orchestrator dispatches the items in the order `fragments ŌåÆ nodes ŌåÆ attributes ŌåÆ links`:
   - **fragments:** for each `{ ref, text }`, call `proposeFragmentHandler` with `{ text, confidence: 1.0, chunk_ids: [the run's chunk id(s) covering the fragment's line] }`. The handler returns `{ ok: true, result: { fragment_id, status, validation_outcome } }` or a rejection envelope. The orchestrator records `ref -> fragment_id` only on `ok: true`.
   - **nodes:** for each `{ ref, node_type, name, node_id?, aliases? }`, either (a) `node_id` pin present ŌåÆ look up the active node by id (rejection envelope `RESOURCE_NOT_FOUND` when the id resolves to no row, or `VALIDATION_INVALID_FORMAT` when it exists but `status != 'active'` ŌĆö P2.1 namespaced; the pin path skips resolution intentionally), or (b) call `proposeNodeHandler` with `{ node_type, name, aliases }` and use the returned `node_id` (any `resolution` outcome ŌĆö `matched_existing` / `created_new` / `needs_review` ŌĆö is forwarded verbatim to the report). The orchestrator records `ref -> node_id` only on `ok: true`.
   - **attributes:** for each `{ node_ref, key, value, evidence_ref, valid_from?, valid_from_basis? }`, resolve `node_ref` and `evidence_ref` via the maps; if either is missing (the referenced item failed earlier), the attribute is **cascaded** ŌĆö see the cascade rule below. Otherwise call `proposeAttributeHandler` with `{ node_id, key, value, confidence: 1.0, fragment_ids: [evidence_id], valid_from?, valid_from_basis: caller-supplied or 'stated' }` and forward the BR-27 outcome verbatim.
   - **links:** mirror of attributes ŌĆö `proposeLinkHandler({ source_node_id, link_type, target_node_id, confidence: 1.0, fragment_ids, valid_from?, valid_from_basis })`.
4. **Cascade rule (the only genuinely new business logic).** An item whose ref dependency has failed (the referenced fragment / node was rejected at its own step) is **skipped** with a synthetic per-item report entry `{ ref, kind, status: 'dependency_failed', reason: <ref-of-missing-dependency> }`. The skip is silent vis-├Ā-vis the database: no `propose_*` call is issued for the cascaded item, no `tool_call` row is written for it (the handler did not run; BR-23 is a writer invariant, not a writer-or-cascade invariant). The orchestrator log records the cascade at INFO level for forensics.
5. **Close the run.** After all items dispatch, the orchestrator calls `closeLlmRunRow(llm_run_id, 'completed')`. The terminal status is **always `completed`** on the directed path ŌĆö a `propose_*` rejection on a single item is an item-level outcome (`rejected` in the report), not a run-level fatal failure. The orchestrator never closes the run as `failed` on this path, because there is no Anthropic call that can fatally fail (BR-29 fatal-failure path is unreachable here). The only way the run lands `failed` on this path is the catastrophic catch-all of step 2 (intake transaction aborted) ŌĆö and in that case the run row never opens at all, so there is no `failed` row written. **Concretely: every successful intake on the directed path produces a `completed` run.**
6. **Envelope.** The handler returns `{ ok: true, result: { outcome: 'ingested', raw_information_id, llm_run_id, chunk_count, run: { ..., affected_nodes? }, report: ItemReport[], summary: { fragments, nodes, attributes, links, accepted, consolidated, superseded_previous, needs_review, uncertain, disputed, rejected, error, dependency_failed } } }` ŌĆö the per-item report has one entry per input item (preserving caller order across the four arrays), and the summary aggregates the eight `validation_outcome` counters of BR-12 plus the directed-only `dependency_failed` counter. `result.run.affected_nodes` is populated synchronously per BR-33 (the directed orchestrator collects the same affected-node map as the LLM orchestrator).

**Cascade rule details (dependency_failed).** The exhaustive cascade triggers are:
- An `attribute` whose `node_ref` is not in the `ref -> node_id` map at dispatch time.
- An `attribute` whose `evidence_ref` is not in the `ref -> fragment_id` map at dispatch time.
- A `link` whose `source_ref` or `target_ref` is not in the `ref -> node_id` map at dispatch time.
- A `link` whose `evidence_ref` is not in the `ref -> fragment_id` map at dispatch time.
A node item with an explicit `node_id` pin that fails the active-node lookup also triggers cascades on any attribute / link that referenced its `ref`. Cascade is **per-item**, not transitive: a cascaded item does NOT cascade further (the cascaded item never registers a ref it did not have anyway). The report entry's `reason` carries the **first** missing dependency ref encountered (deterministic by the dependency-check order).

**Audit (BR-23 unchanged).** Every dispatched `propose_*` call writes its own `tool_call` row via `runIngestHandler` ŌĆö exactly as the four REST mirrors and the four LLM-driven MCP tools do (BR-23, BR-28). Cascaded items write NO `tool_call` row (no handler runs). The directed orchestrator itself writes no top-level `tool_call` row for the `ingest_directed` invocation ŌĆö same as BR-30 / BR-31, which are also not `propose_*` writers. The audit trail of an `ingest_directed` run is therefore the union of N `tool_call` rows (one per dispatched `propose_*`) plus the `LLMRun` row (`model='directed'`, `prompt_version='directed-v1'`) ŌĆö the count of `tool_call` rows is strictly less than the count of input items when any cascade fires.

**Transactional boundary (BR-19 unchanged).** Each dispatched `propose_*` opens its own transaction; the directed orchestrator does NOT wrap the whole dispatch in a single transaction (same shape as BR-26 ŌĆö one TX per tool call). A `propose_*` rejection on item K does NOT roll back the K-1 already-accepted items; they remain persisted, exactly as in the LLM-driven extraction. This matches the user-facing semantics promised by the per-item report: partial success is a first-class outcome.

**Re-affirmation semantics.** Because step 2 stamps the synthesised content with a per-call nonce, the `RawInformation` / `LLMRun` of two successive `ingest_directed` calls are **always distinct** (decision 3 of the approved plan). However, the per-item `propose_*` calls inside the second run hit the standard graph-consolidation path of BR-27: a `propose_link` / `propose_attribute` that re-affirms a previously accepted assertion lands as `consolidated` (provenance accumulates on the existing link / attribute, no second vigent row is inserted). This preserves ┬¦18 "re-affirmation consolidates, never duplicates" at the graph layer while keeping every directed call uniquely audit-addressable at the run layer.

**Distinct from BR-30 (`ingest_document`).** BR-30 takes raw `content` and asks the **server's** Anthropic to extract structured knowledge from it; BR-34 takes the structured payload directly and asks the server to persist it deterministically. BR-30 is for callers that have unstructured documents and want extraction; BR-34 is for callers that already know what they want to ingest (typically the chat LLM, which structures the turn context client-side). The synchronous, blocking-but-bounded return semantics are otherwise identical. Defaults differ: BR-30 reads `model` / `prompt_version` from the args (with `INGEST_MODEL` / `DEFAULT_PROMPT_VERSION = 'v3'` fallback); BR-34 ignores any caller-supplied model / prompt-version (there is no Anthropic call to parameterise) ŌĆö the run carries the sentinels `'directed'` / `'directed-v1'` unconditionally.

**Distinct from withdrawn BR-32 (`start_async_ingestion`).** BR-32 fired an LLM extraction asynchronously; BR-34 runs **no LLM** and returns synchronously. The chat agentic loop's prior pattern (call `start_async_ingestion`, poll `get_ingestion_status` for `affected_nodes`) is collapsed into a single call (`ingest_directed` returns `affected_nodes` inline). There is no detached-promise lifecycle, no orphan-on-restart hazard, no flag-guarded background path.

**Distinct from BR-21 (`propose_*`).** BR-21 governs the four writer tools that operate INSIDE an existing `running` `LLMRun` (caller supplies `llm_run_id`); BR-34 **creates** the run. BR-34 does NOT take `llm_run_id` and is not subject to `assertRunIsRunning` at the handler level. Internally, every dispatched `propose_*` call respects BR-21 verbatim ŌĆö the orchestrator IS the run owner, so `assertRunIsRunning` is trivially satisfied.

**Chat integration.** This tool is the back-end half of the chat-driven ingestion surface (`chat.back.md` v4.0.0). The chat module resolves it through the **standard catalog path** ŌĆö `CHAT_INGEST_TOOL_NAMES = ['ingest_directed']` is appended to the chat tool catalog when `env.CHAT_INGEST_ENABLED === true` (boot-only flag, default `false`). When the flag is off, the tool is absent from the chat agent's tool set (the chat agent does not advertise an ingestion capability); the tool is **still registered** on the `ingest` toolset itself ŌĆö an external MCP client (`mcp-remote`, Claude Desktop, the local stdio path) can always call it. The previous OPTIONAL-resolver seam (`mcp.getTool('ingest', 'start_async_ingestion')`, `buildIngestDispatcher` injection, the `ingestDispatcher` special-case in `chat-agent.service.ts`) is **removed** alongside BR-32 ŌĆö `ingest_directed` is a normal catalog tool, not a special case.

**Rollout flag.** Unlike BR-32, `env.CHAT_INGEST_ENABLED` no longer gates the **registration** of the tool on the `ingest` toolset ŌĆö the directed tool is always registered (it is deterministic and has no operational risk worth a registration-time kill switch). The flag now only gates the chat module's catalog inclusion (boot-only; see Chat integration). Flipping the flag still requires a BFF restart, same as before.

**Defaults / behaviour matrix.**
| Aspect | BR-34 (`ingest_directed`) |
|---|---|
| LLM call on server | **None** |
| Run `model` | `'directed'` (sentinel) |
| Run `prompt_version` | `'directed-v1'` (sentinel) |

**Error returned:**
- Zod failure ŌåÆ `{ ok: false, error.code: <VALIDATION_REQUIRED_FIELD | VALIDATION_INVALID_FORMAT | VALIDATION_OUT_OF_RANGE> }` (P2.1 Zod-discriminated; no run opens, no `tool_call` row).
- Intake failure (step 2) ŌåÆ `{ ok: false, error.code: 'SYSTEM_SERVICE_UNAVAILABLE' }` when the database is unavailable (`isPgUnavailable`), else `{ ok: false, error.code: 'SYSTEM_INTERNAL_ERROR' }` (P2.1 namespaced) with a generic message (raw `err.message` is logged server-side only).
- Per-item rejection (any `propose_*` rejection envelope) ŌåÆ the rejection is forwarded verbatim into `result.report[i]`; the top-level envelope stays `{ ok: true, ... }`. The run completes; the rejection is observable via the report and via the run's `tool_call` rows.
- Per-item cascade (dependency_failed) ŌåÆ synthetic report entry `{ ref, kind, status: 'dependency_failed', reason }`; no `tool_call` row; the top-level envelope stays `{ ok: true, ... }`.
- All envelopes are surfaced as MCP 2025-06-18 `content` / `isError: true` only for the top-level failure cases (Zod, intake); `isError: false` (HTTP `200` from the SDK kernel) on the success path even when individual items rejected. **No new error code** ŌĆö every code is already registered in `docs/specs/_global/error-codes.md`.

## 4. State Machine (ST)

### ST-01 -- LLMRun lifecycle (ST-LR of `ingestion.spec.md` ┬¦5.1)

```
            ingestRawInformation
                    |
                    v
               [running] --runLlmExtraction success---> [completed]   (terminal)
                    |        (BR-26 step 6)
                    |
                    +--runLlmExtraction fatal failure--> [failed]
                    |        (BR-26 step 7)
                    |
                    +--close err (legacy direct path)---> [failed]
                                       |
                                       +-- retryLlmRun --> [running]
                                              (BR-10)
```

| From | To | Event | Guard | UC |
|------|----|-------|-------|----|
| (nothing) | running | `ingestRawInformation` creates a new run | `idempotency_key` is not already held by a `non-failed` run; UNIQUE constraint | UC-01 |
| running | completed | `runLlmExtraction` finishes all chunks without fatal failure | BR-26 step 6 ŌĆö single UPDATE setting `status='completed'`, `finished_at=now()` | UC-12 |
| running | failed | `runLlmExtraction` catches unhandled exception | BR-26 step 7 ŌĆö single UPDATE setting `status='failed'`, `finished_at=now()` | UC-12 |
| running | failed | legacy close path (close-from-MCP, retained for backward compat) | service sets `finished_at = now()`, `status = 'failed'`; DB `CHECK (status='running') = (finished_at IS NULL)` enforces simultaneous update | UC-07 |
| failed | running | `retryLlmRun` | atomic `UPDATE ... WHERE status = 'failed'`; in same TX orphan `proposed` fragments ŌåÆ `rejected` | UC-06 |
| completed | ŌĆö | ŌĆö | terminal ŌĆö `retryLlmRun` returns `409 BUSINESS_RUN_NOT_RETRYABLE`; `runLlmExtraction` returns `409 BUSINESS_RUN_NOT_RUNNABLE` | ŌĆö |
| running | ŌĆö | `retryLlmRun` while still running | rejected ŌåÆ `409 BUSINESS_RUN_NOT_RETRYABLE` (BR-11) | UC-06 alt 2b |
| running | ŌĆö | `runLlmExtraction` while already running | rejected ŌåÆ `409 BUSINESS_RUN_NOT_RUNNABLE` (BR-26 pre-check) | UC-12 |

**Invalid transitions:** any transition from `completed` triggered by `retryLlmRun` or `runLlmExtraction` is rejected at the service layer. The DB `CHECK (status = 'running') = (finished_at IS NULL)` rejects any UPDATE that desyncs the two columns.

### ST-02 -- InformationFragment lifecycle (ST-IF of `ingestion.spec.md` ┬¦5.2)

Unchanged from v1.1.0. The new orchestrator (BR-26) drives the same `proposed ŌåÆ accepted | rejected` transitions through the same service functions; consolidation (BR-27) is the path that promotes `proposed ŌåÆ accepted` via the cited fragments accumulating provenance.

## 7. Known Technical Constraints

- **Single-instance assumption for advisory locks.** Unchanged ŌĆö see v1.1.0.
- **No row-level security.** Unchanged.
- **10 MiB body cap is per request, not per document.** Unchanged.
- **Chunker is in-process, synchronous, single-threaded.** Unchanged.
- **`Intl.Segmenter('pt')` requires Node.js built with full ICU.** Unchanged.
- **Catalog data (`NodeType`, `LinkType`, `AttributeKey`, `LinkTypeRule`) is read-only at runtime.** Unchanged; cached in `CatalogSnapshot` at boot; consumed by both the validators (BR-13..BR-17) and the prompt builder (BR-26 step 5a). A migration that adds catalog rows requires a BFF restart to surface them.
- **`tool_call.arguments` and `tool_call.result` are stored verbatim.** Unchanged ŌĆö but the new in-process orchestrator (BR-26) routinely produces tool-call counts in the tens-to-low-hundreds per run, which fits comfortably within the ┬¦16 ceiling.
- **PostgreSQL `UNIQUE` violation detection relies on SQLSTATE `23505` and constraint name.** Unchanged; this revision adds awareness of `knowledge_link_current_dup_guard` and `node_attribute_current_dup_guard` ŌĆö those should now only fire on a programming error (consolidator skipped, BR-27) **or** on a real concurrent race against the `FOR UPDATE` lock release. The consolidator's `23505` catch path runs the lookup-and-decide step **once** more before giving up ŌĆö see BR-27 Dup-guard catch path. The error returned on a second `23505` MUST distinguish a real race from a deterministic miss; the legacy message (`"hit dup-guard twice; concurrent contention not resolvable"`) is forbidden because it conflated the two.
- **Re-affirmation of a multi-current link / attribute does NOT require matching `valid_from`.** For `allows_multiple_current = true` link types (e.g. `holds_role`) and `allows_multiple_current = true` attribute keys, the schema's partial UNIQUE dup-guard is scoped by `(source, link_type, target)` / `(node, attribute_key, value)` ŌĆö `valid_from` is deliberately NOT part of the key. The consolidator (BR-27 sub-case (i)) treats a re-asserted link / attribute with the same target / value as `consolidated` regardless of whether the new proposal's `valid_from` matches the vigent row's `valid_from`. The typical trigger is the per-document `received` fallback (FR-001 in `temporal.ts`): two documents received on different days re-assert the same fact, the LLM provides no `valid_from`, the fallback yields a different `valid_from` per document, and ┬¦18 still requires consolidation. The vigent row's `valid_from` is preserved; promotion of a stronger evidence basis (e.g. `received ŌåÆ stated`) is **out of scope** for this revision (┬¦8).
- **`unaccent()` is STABLE in Postgres.** Unchanged.
- **Neon Auth JWKS keys are EdDSA by default.** Unchanged.
- **Anthropic API is the only LLM provider (BR-29).** Switching providers requires a code change and a new release. The `ANTHROPIC_API_KEY` is the only LLM-side secret; it is loaded once at boot and never propagated to the request context.
- **`runLlmExtraction` is synchronous; the HTTP request stays open for the full run.** v1 acceptable per ┬¦16 (LLM-bound, minutes per document is fine at the v1 scale of hundreds of documents). Frontend / external callers MUST set their own client timeout to at least the expected per-document budget. A future evolution is a job queue; out of scope for v1.
- **Anthropic `stop_reason = 'pause_turn'` requires loop continuation without modifying messages.** The orchestrator implementation must handle this verbatim per BR-26 step 5b; failure to do so would silently truncate extractions. Covered by the orchestrator unit test (stubbed Anthropic client emitting a `pause_turn` then an `end_turn`).
- **No DDL change in this revision.** The schema (`migrations/0001_init.sql`) already supports entity resolution (`node_alias.alias_norm` GENERATED, `node_alias_norm_trgm_idx` GIN, `entity_match_review` table, `knowledge_node.status` / `merged_into_node_id`, `norm()` / `immutable_unaccent()` functions, partial dup-guard indexes). If a future tweak to thresholds or indexes requires DDL, it follows the "Database Changes Require Explicit Approval" rule of CLAUDE.md.
- **Entity-resolution thresholds (`MATCH_STRONG = 0.85`, `MATCH_FLOOR = 0.55`) are not configurable per call** (BR-25). They live in `entity-resolution.service.ts`; tuning requires a code change. The ┬¦16 metrics (acceptance rate, `needs_review` rate) are the calibration input.
- **`merged_into_node_id` path compression must happen on write** (CLAUDE.md "Known Gotchas"). The resolver hits `knowledge_node` filtered by `status = 'active'`, so merged nodes are naturally excluded from candidates; the merge service (`curation/service/merge.service.ts`) is responsible for repointing inbound edges to active survivors ŌĆö no change in this revision.
- **The orchestrator does NOT hold a `pg` connection across turns.** Each tool-use invocation calls the corresponding service which acquires its own connection (BR-19). Otherwise the pool of `max=10` would be exhausted by a single long run.

## 8. Out of Scope

- **Embeddings / vector search.** Permanent non-goal (┬¦20.1, A24, CLAUDE.md "Anti-patterns").
- **System-time travel** (┬¦5.3 query c, A25) ŌĆö permanently deferred.
- **Multi-tenant / `User` entity** (┬¦2.3, ┬¦20.3, A20) ŌĆö permanent non-goal.
- **Catalog mutation API.** Migration-only (┬¦12).
- **Event bus / message queue.** Database is the integration boundary (┬¦2.2 "store ├║nico").
- **Rate limiting / quota.** Single-owner; no per-tenant quota required in v1.0.0.
- **Async / queued ingestion or extraction.** `ingestRawInformation` and `runLlmExtraction` are both synchronous in v1; future worker offload is out of scope.
- **Re-chunking of existing documents.** Bumping `chunking_version` requires a new migration; not exposed in v1.
- **Retrieval (┬¦7)** ŌĆö owned by future `retrieval` domain.
- **Curation (┬¦10)** ŌĆö owned by future `curation` domain. The `entity_match` queue produced by BR-25 is consumed by the **existing** `resolveEntityMatchService` / `performMerge` (`curation/service/merge.service.ts`), with zero curation-side change required by this revision.
- **Compliance deletion (┬¦11)** ŌĆö owned by future `compliance` domain.
- **Alternative LLM providers.** Anthropic is the sole integration (BR-29); switching providers is a code-level change, not configuration.
- **Per-run wall-clock cap or cost cap on the Anthropic call.** v1 trusts the LLM to terminate the loop at `stop_reason='end_turn'`; the orchestrator is otherwise patient. A future evolution may add a `max_turns_per_chunk` cap or a per-run token-budget; out of scope for v1.
- **Re-running an extraction from a partial state.** Retry (`retryLlmRun` + `runLlmExtraction`) starts the extraction from the first chunk again; the consolidation flow (BR-27) absorbs the re-runs by `consolidated` outcomes. A future evolution may add a "resume from chunk N" mode using the existing `tool_call` history; out of scope for v1.
- **Schema migration in this revision.** No DDL; if any future entity-resolution/consolidation tweak requires DDL, it is handled in its own change (subject to CLAUDE.md's "Database Changes Require Explicit Approval" rule).
- **DDL for `llm_run.affected_nodes_snapshot`.** BR-33 declares an OPTIONAL `affected_nodes` field on the run's read surface; v1.3.0 satisfies the contract via an in-process LRU cache keyed by `llm_run_id` with derived-from-`tool_call` fallback. Out of scope for v1.3.0.
- **Promotion of `valid_from_basis` on consolidation.** ┬¦6.5 declares an evidence-basis precedence (`stated > document > received`). A future evolution may, on consolidation of a vigent row whose `valid_from_basis = 'received'` with a new proposal whose `valid_from_basis = 'stated'`, update the vigent row's `valid_from` and `valid_from_basis` to the stronger basis (instead of merely accumulating provenance). This revision does **not** implement that promotion ŌĆö re-affirmation preserves the vigent row's `valid_from` and `valid_from_basis` as-is in every sub-case of BR-27. The bug closed by this revision is strictly the deterministic mis-rejection of multi-current re-affirmation; the basis promotion is a separate change.
